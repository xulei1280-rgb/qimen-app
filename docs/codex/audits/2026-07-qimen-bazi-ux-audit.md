## 1. 执行摘要

当前产品不是“功能不够”，而是已经具备相当完整的奇门、八字和 AI 工作流之后，开始暴露三类长期使用问题：

1. **盘面身份和页面状态不够稳定。** 奇门输入参数变化后，旧盘仍可能继续被保存或送入 AI；奇门和八字的 AI 请求也可能在切换记录后返回到错误的当前状态。这类问题优先级高于视觉优化。
2. **本地记录可靠性不足以支撑数百案例。** 八字记录存在静默截断到 50 条的风险；解析失败、空间不足、重复保存、误删除和“更新还是另存”均缺少完整保护。
3. **专业信息没有充分分层。** 八字结果页在手机端约 4868px、桌面端约 3287px，核心内容导航分别约在 1451px 和 1334px 后才出现；奇门在部分平板宽度和 320px 窄屏会横向溢出。内容正确，但查找、对照和复盘成本高。

最重要的产品方向不是重做命理逻辑，而是建立四个稳定基础：

- 一张盘、一次 AI 请求、一条记录之间有明确且不可漂移的身份绑定。
- 保存、更新、另存副本、自动历史、删除和恢复的语义清楚。
- 同一份正确排盘结果按“概览—标准盘—专业依据”逐层展示。
- 奇门和八字共用记录、搜索、标签、备份、删除撤销和状态反馈；盘面结构与专业交互保持独立。

### 当前已经做得好的部分

- 奇门九宫盘视觉辨识度高，在 390px 手机宽度仍能保持基本可读；当前、上一时辰、下一时辰入口顺手。
- 奇门已经有真太阳时预览、问事类型、同主题追问保护，以及 AI 问答逐问折叠、删除和一键收起。
- 八字输入页清楚，真太阳时修正过程透明；结果默认进入格局喜用，证据、条件和不确定性表达较完整。
- 八字已有大运、流年、流月选择链的基础，AI 回答也支持逐条折叠和删除。
- 本轮浏览器审查的主要流程未发现运行时控制台错误。

### 必须保留、不能被 UX 改造破坏的部分

- 现行奇门、八字排盘结果和全部正确算法。
- 真太阳时、节气、四柱、格局喜用、大运流年和 AI 推理逻辑。
- 现有 `qimen_auto_history`、`qimen_history`、`bazi_records` 数据的可读取性。
- 奇门九宫盘与八字四柱/运势表各自适合专业判断的视觉语义。
- 用户标签、笔记、收藏、排序等管理信息默认不进入 AI 上下文。

## 2. 当前产品结构

### 2.1 当前奇门流程

```text
输入时间/地点/真太阳时/问题
        ↓
起盘并显示状态摘要
        ↓
九宫盘 ── 点击宫位 → 侧栏/下方详情
        ↓
AI 首次分析 → 继续追问 → 折叠/删除答案
        ↓
保存、复制、导出、自动历史/手动记录
```

优点是“起盘—看盘—问事”在一个工作区内完成；问题是输入状态、已生成盘面、AI 会话和已保存记录之间缺少显式的当前对象模型。

### 2.2 当前八字流程

```text
出生资料输入
      ↓
生成八字结果
      ↓
长结果总览/四柱表
      ↓
格局喜用 | 大运流年 | AI 分析等内容区
      ↓
保存、编辑、删除、重新打开记录
```

优点是专业结果充分；问题是内容导航出现太晚、返回与恢复当前结果路径不足，保存动作又藏在 AI 区域，导致“看盘”和“管理当前命盘”割裂。

### 2.3 当前记录与数据边界

| 模块 | 当前存储 | 当前用途 | 已识别限制 |
|---|---|---|---|
| 奇门自动历史 | `qimen_auto_history` | 最近起盘/自动记录 | 上限约 60 条；与正式保存语义容易混淆 |
| 奇门正式记录 | `qimen_history` | 用户主动保存 | 打开后继续 AI 可能写入自动历史；保存可能重复创建 |
| 八字记录 | `bazi_records` | 命盘与 AI 历史 | 每次写入存在 `slice(0, 50)` 静默丢弃旧记录风险 |

三份数据应继续物理独立，未来只在读取层做统一索引，避免一次性迁移破坏兼容性。

### 2.4 已完成的界面证据

| 视口/场景 | 观察结果 |
|---|---|
| 八字手机 390×844 | 结果页约 4868px 高，主内容标签约在 1451px 后；用户需先越过多组重复摘要和完整四柱表 |
| 八字桌面 1440×900 | 结果页约 3287px 高，主内容标签约在 1334px 后；宽屏没有充分转化为更短的查阅路径 |
| 奇门手机 390×844 | 九宫盘基本可读；完整页面约 2220px，侧栏信息和记录入口被推到盘面、工具、问题、AI 之后 |
| 奇门窄屏 320px | 页面 `clientWidth≈305`、九宫盘宽约 340px、文档 `scrollWidth≈358`，存在真实整页横向溢出 |
| 奇门平板/小桌面 1024px | 双列最小宽度之和超过可用空间，而响应式断点到 900px 才切换，约 901–1085px 是高风险区间 |

这些数字用于判断信息层级和响应式风险，不代表建议锁死具体像素值。

| P2-11 | 八字出生时辰不确定时，页面容易让用户输入一个看似确定的值；但自动反推又风险极高 | 后续结论被赋予虚假的精确性 | 只增加“时辰待确认”标记和醒目免责声明；允许保存候选记录或备注，不自动根据真人经历反推时辰 | 身份条显示“时辰待确认”，专业结论默认折叠风险提示 | 结果顶部保留待校时状态和候选说明 | 中 | 排盘：不改变任何推算；AI：明确不确定性；历史：增量标记 | P2 |
| P2-12 | GitHub Pages 发布新版本后，手机缓存可能继续显示旧页面；强制刷新又可能丢当前草稿 | 用户误以为功能没有发布，或更新时丢工作 | 显示当前版本和非阻断“发现新版本”；刷新前保存可过期草稿；不自动强刷正在工作的页面 | 顶部更新条可稍后处理 | 设置/页脚显示版本与更新时间 | 中 | 排盘：不改；AI：不改；历史：不改 | P2 |

## 4. 用户未提出但值得关注的问题

### 4.1 “能返回”不等于工作连续

长期用户经常在盘面、AI、记录和输入页之间往返。当前缺少“当前工作对象”的概念，导致离开结果页、打开另一记录或刷新页面后，很难判断自己正在编辑哪一盘、是否已保存、AI 正在回答哪一盘。P0-03、P0-05、P0-06 和 P2-08 应被视为同一套工作区状态设计，而不是四个孤立补丁。

### 4.2 时间边界需要风险反馈，而不是增加算法选项

靠近日界、节气交接或真太阳时跨时辰时，用户更需要看到“输入值、修正值、最终采用值以及临界提示”。这不要求新增流派或允许随意切换规则，只需把现有计算结果做成可核对回执。对应 P1-12。

### 4.3 长 AI 对话会从“回答”变成“资料库”

十几到几十问后，简单纵向卡片不再够用。会话需要问题编号、目录、当前项自动定位、草稿保护和复制/重试；但不建议首批做复杂分支树或全文向量搜索。对应 P1-05、P1-14。

### 4.4 专业案例需要“事实、笔记、AI”三层分离

用户记录的生平事实、自己的判断笔记、AI 回答具有不同可信度和用途。未来即使增加笔记，也必须：

- 默认不进入 AI 上下文；
- 不反向参与格局、喜用或任何规则制定；
- 与原始盘面快照和 AI 会话分区展示；
- 允许匿名导出时分别排除。

对应 P2-03、P2-04。

### 4.5 旧记录需要来源可解释

规则或展示版本升级后，旧卡片若直接显示“格局等级/评分”，用户会把缓存旧结论误当成当前引擎结果。因此记录卡只显示稳定事实字段；需要专业结论时打开记录再明确选择“查看历史快照”或“用当前版本复核”。对应 P2-07。

### 4.6 搜索越多不一定越好

不建议默认索引完整 AI 正文。全文回答通常很长、含敏感信息且噪声高，会降低结果相关性并增加本地索引体积。首批搜索稳定字段、问题摘要和用户标签；会话内搜索作为独立能力放在 P2。

### 4.7 手机端的隐性长期问题

- 软键盘弹起后固定作曲器可能遮住回答或发送按钮。
- 浏览器返回手势与横向表格/左右切盘手势可能冲突。
- 底部安全区、地址栏伸缩和 320px 小屏会放大固定高度问题。
- 长页面重渲染后滚动锚点跳动，折叠状态和草稿容易丢失。
- 仅靠 hover 的帮助说明在触屏上不可发现。

这些问题应通过布局、焦点和状态保存解决，不应再增加隐藏手势。

### 4.8 奇门与八字哪些共用、哪些独立

| 可以共用 | 必须独立 |
|---|---|
| 全局模块切换、记录中心、搜索、排序、标签、收藏、最近使用、备份、删除撤销、保存/脏状态、AI 问答卡和错误反馈 | 奇门九宫布局、宫位详情、时辰前后切换、问事类型与起局摘要 |
| AI 请求绑定、匿名导出、草稿恢复、可访问性规范 | 八字四柱表、格局喜用证据、大运→流年→流月选择链与专业密度 |
| 统一“盘面身份条”的交互模式 | 身份条中的字段、校盘顺序和专业术语 |

共用的是产品壳层和可靠性能力，不是把两种盘面做成同一种卡片。

### 4.9 原始问事是奇门记录的身份，不是可被追问覆盖的输入框

奇门适合围绕一次起局时的具体事件持续追问。原始问事一旦进入首次 AI 分析，应固定在身份条和记录索引中；继续追问不能改写标题或 `question`。若用户要换一个独立事件，应明确新起一局。八字则不同：同一命盘长期围绕事业、感情、健康和不同运年分析，后期可以有多个主题会话。对应 P0-10、P1-16。

### 4.10 “默认设置”和“本盘采用值”必须分开

真太阳时偏好、地点预设、实际经度、输入时间和最终采用时间分别属于用户偏好、表单状态和盘面事实。它们不能靠同一个开关或地点文字隐含表达。尤其是奇门当前存在偏好已保存但初始化仍可能关闭、加载经度却未同步地点名称的风险。对应 P0-09、P1-12、P1-15。

### 4.11 本地应用也需要明确的数据位置

GitHub Pages 同步的是发布代码，不是浏览器 localStorage。电脑、手机、不同浏览器和无痕窗口各自独立；浏览器清理或系统回收存储也可能删除记录。页面必须在首次保存和记录中心明确“仅此设备”，并把可读报告与可恢复备份区分开。对应 P1-17、P1-19。

### 4.12 数据、规则推导和 AI 解释必须有来源标签

四柱、局式等硬数据，与格局/关系规则推导、AI 自然语言解释的可靠性和可复核方式不同。“本地校验未发现冲突”只能表示有限基础字段核对，不能形成“整篇 AI 已验证”的权威暗示。对应 P1-18。

### 4.13 页面更新也属于数据保护场景

静态站点发布后，移动浏览器缓存未必立即刷新；如果简单要求用户强刷，又会威胁未保存盘面和 AI 草稿。后期应有版本显示、非阻断更新提示和刷新前草稿保护。对应 P2-12。

### 4.14 当前完全未覆盖或覆盖很弱的高频需求

以下不是新增孤立建议，完整实现字段以对应主台账为准。

| 需求 | 典型使用场景 | 对应台账 |
|---|---|---|
| 不进入 AI 的个人笔记/待复核项 | 专业用户记录线下反馈、自己的判断和以后要复核的点 | P2-03 |
| 八字多主题 AI 会话 | 同一命盘分别讨论事业、感情、健康和某步大运 | P1-16 |
| 长会话目录、搜索和上下文范围 | 几十问后定位旧结论、避免把全部历史持续送入 AI | P1-14、P1-16 |
| 电脑与手机间无登录搬运 | 在电脑保存后需要在手机继续，又不引入账号系统 | P1-17、P1-19 |
| 可读报告与可恢复备份 | 一份用于分享阅读，一份用于恢复全部记录 | P1-19、P2-04 |
| 版本和来源追溯 | 规则升级后解释旧案例为何显示不同 | P2-07 |
| 时辰待确认状态 | 不知道准确出生时辰，但不允许自动反推 | P2-11 |
| 手工关联奇门事件与八字命盘 | 同一对象有长期命盘和多个具体事件盘 | P2-10 |
| 本机存储健康/最近备份 | 数百条案例后确认容量、损坏和备份状态 | P0-08、P1-17 |
| 更新提示与刷新前草稿保护 | GitHub Pages 已发布但手机仍缓存旧版 | P2-08、P2-12 |

## 5. 保存记录管理专项设计

### 5.1 设计目标

记录中心要在 10 条时不显笨重、在 300–1000 条时仍可查找，并且不要求先迁移现有 localStorage。第一原则是数据安全，第二原则是打开后状态连续，第三原则才是高级筛选。

### 5.2 兼容式统一索引

保留三份原始数据作为事实来源；在内存中适配为统一列表，可选增加可重建的旁路元数据键（例如 `qimen_app_record_meta_v1`），但不在启动时重写旧记录。

| 统一字段 | 用途 | 兼容策略 |
|---|---|---|
| `uid` | 前端统一身份 | 由模块、集合和原始 ID 组合，不替换原 ID |
| `module` | `qimen` / `bazi` | 适配时生成 |
| `collection` | 奇门自动/正式、八字正式 | 保留来源语义 |
| `sourceId` | 原始数字或字符串 ID | 原样保留 |
| `title` / `person` | 标题和人物分离 | 旧 `name` 不删除，缺失项标为旧记录 |
| `chartDate` / `pillars` | 起局/出生时间与四柱 | 只读取已有稳定事实 |
| `question` | 奇门原问题摘要 | 最多两行参与搜索，不索引完整 AI |
| `systemTag` / `customTags` | 系统分类和用户标签 | 自定义标签存旁路元数据，默认不进 AI |
| `createdAt` / `updatedAt` / `lastOpenedAt` | 排序 | 旧记录按可用字段逐级回退 |
| `pinyinFull` / `pinyinInitials` / `alphaBucket` | 中文和 A–Z 定位 | 索引可重建，支持多音字人工覆盖 |

旁路元数据丢失时只损失标签、收藏和最近使用，不应影响原始命盘。孤立元数据可忽略，未知原字段必须保留。

### 5.3 搜索

首批统一搜索：

- 姓名/称呼、记录标题；
- 起局/出生日期、创建日期；
- 四柱，可支持“甲子 乙丑”与“甲子乙丑”；
- 奇门原始问题摘要和问事类型；
- 系统分类、自定义标签；
- 中文全拼和拼音首字母。

不首批索引：完整 AI 正文、格局评分、可能随规则版本变化的缓存结论。

### 5.4 拼音 A–Z 分类和手机快速定位

- 仅在“姓名 A–Z”排序时显示字母分组，避免排序逻辑冲突。
- 中文按本地拼音索引，英文取首字母，无姓名或无法归类进入“#”。
- 多音字允许在元数据中人工覆盖，不能改姓名原文。
- 手机右侧显示可拖动字母轨道，只突出存在数据的字母；同时提供可访问的“跳到字母”下拉菜单。
- 桌面使用分组标题和左侧字母索引，键盘输入字母可跳转。

### 5.5 排序与筛选

排序：

1. 最近使用：`lastOpenedAt`，缺失时按 `updatedAt → createdAt → 原数组顺序` 回退；
2. 创建时间：新到旧/旧到新；
3. 姓名：A–Z/Z–A；
4. 盘面日期：近到远/远到近。

筛选：

- 奇门/八字；
- 奇门自动历史/正式保存；
- 问事类型；
- 自定义标签；
- 日期范围；
- 有/无 AI；
- 收藏（P2）。

手机筛选使用底部动作表，桌面使用可收起侧栏。筛选结果、滚动位置和当前排序在返回记录中心时恢复。

### 5.6 记录卡片

奇门卡片固定展示：

- 标题/姓名、自动或正式标记；
- 问事类型、起局时间、四柱；
- 原问题最多两行；
- AI 问答数、最近使用时间、标签。

八字卡片固定展示：

- 标题/姓名、性别；
- 出生时间、四柱、地点；
- 真太阳时状态、AI 问答数、最近使用时间、标签。

不在八字卡片展示格局等级或评分，以免旧缓存和当前规则混淆。整张卡可打开，编辑、删除、导出等操作放进至少 44px 的“更多”菜单。

### 5.7 保存、更新、另存和编辑

```text
当前盘未保存
  └─ 保存 → 填标题/姓名/标签 → 创建正式记录

从正式记录打开
  ├─ 元数据未变 → 显示“已保存”
  ├─ 仅标题/标签变 → 更新此记录
  └─ 排盘输入或盘面身份变 → 默认另存副本
                           └─ 覆盖原记录需明确确认

奇门自动历史
  └─ “转为正式记录” → 保留来源，生成正式记录身份
```

修改姓名/称呼时要说明“将影响后续 AI 的人物上下文”；修改标签和收藏不进入 AI。

### 5.8 重复保存

重复指纹只用于提醒，不能自动合并：

- 奇门：采用时间 + 真太阳时/经度 + 四柱 + 原始问题；
- 八字：输入/采用时间 + 地点 + 四柱 + 人物。

检测到相似记录时提供“更新已有 / 另存副本 / 取消”，并显示两条记录的日期、标题和 AI 数量差异。

### 5.9 边界状态

| 场景 | 推荐体验 |
|---|---|
| 完全无记录 | 说明“记录只保存在当前浏览器”，提供“新建奇门/新建八字” |
| 搜索无结果 | 保留查询，显示命中的筛选条件，提供“一键清除筛选” |
| 旧记录缺字段 | 标“历史记录·信息不完整”，仍可打开，不自动猜值 |
| JSON 损坏 | 停止覆盖，允许导出原始文本和查看恢复说明 |
| localStorage 空间不足 | 保留当前盘和输入，明确保存未成功，提供备份和清理入口 |
| 删除当前记录 | 盘面继续留在内存，状态改为“未保存”，8–10 秒可撤销 |
| 奇门自动历史到 60 条 | 明示滚动淘汰策略；重要项可先转为正式记录 |
| 重复保存 | 显示相似项，不自动合并 |
| 批量操作 | 首批只做批量标签/导出；批量删除等备份和回收机制成熟后再开放 |

### 5.10 数百条记录后的性能

- 300 条以内可本地增量索引；更大时采用分页或虚拟列表，不无限渲染全部卡片。
- 搜索输入防抖，但卡片打开和最近使用写入不应等待搜索。
- 拼音索引和标签统计只在原始记录变化时重建。
- 备份导出包含原始三键和旁路元数据，并带格式版本；导入先预览数量和冲突，不直接覆盖。

## 6. 手机端专项设计

### 6.1 总体原则

手机端不是把桌面端整体缩小，而是保持盘面语义、减少同时出现的层级。任何复杂横向信息只能在局部容器内滚动，整页必须无横向溢出。

### 6.2 奇门

推荐顺序：

1. 吸顶盘面身份条：时间、局数、真太阳时、保存/脏状态；
2. 九宫主盘；
3. 选中宫位详情（紧邻盘面或底部抽屉）；
4. “盘面 / AI / 记录”可见分段导航；
5. AI 问答与固定作曲器。

改动重点：

- 320px 完整显示九宫，不使用整页缩放或横向滚动；
- 上一时辰/当前/下一时辰按钮达到触控尺寸；
- 宫位抽屉关闭后保持选中宫位与滚动位置；
- 记录中心全屏打开，而不是继续堆在页面底部；
- Key、导出和显示设置放二级菜单；真太阳时关闭时收起经度细节；
- 色彩图例默认收起，但危险/空亡等不能只用颜色表达。

### 6.3 八字

推荐顺序：

1. 吸顶身份条和“返回当前命盘”；
2. 紧凑四柱概览；
3. 可横向滚动的主标签：“原局 / 格局喜用 / 大运流年 / AI”；
4. 当前内容；
5. AI 页固定作曲器。

大运、流年和流月采用逐层选择，只展示当前层的重点和下一层入口。专业全表保留在“完整表”中，使用局部横向滚动、粘住首列/当前列及“向右查看更多”提示。

### 6.4 字号、按钮、滚动和折叠

- 正文和 AI 回答建议 15–16px；元数据不小于约 12px；不要用 9–10px 承载核心判断。
- 触控目标约 44×44px；相邻编辑/删除之间留足间距。
- “全部展开/全部收起”只在存在两组以上折叠内容时显示。
- 新 AI 回答完成后滚到该回答标题，而不是页面底部；用户主动向上阅读时不强制跳动。
- 软键盘弹起时作曲器和发送按钮保持可见，底部留安全区。
- 页面返回保留筛选、折叠、选中宫位/运岁和滚动锚点。

## 7. 桌面端专项设计

### 7.1 奇门工作区

- 安全宽度足够时采用“九宫主盘 2/3 + 上下文侧栏 1/3”。
- 侧栏只放盘面身份、选中宫位和当前 AI/记录状态，不重复完整盘面摘要。
- 小桌面和平板横屏在约 1080px 前后即切单列，实际断点以内容最小宽度测试决定，不固守当前 900px。
- AI 可在侧栏和独立阅读页之间切换；长回答不挤压九宫盘。

### 7.2 八字工作区

- 顶部保持命盘身份条与全局操作；结果标签吸顶。
- 四柱概览、核心结论和证据采用主次双栏；同一结论只保留一个正文来源。
- 大运→流年→流月使用连续选择链，当前列固定突出。
- AI 正文控制行宽，问题目录放侧栏；保存/导出不与发送按钮混在一起。

### 7.3 记录中心

- 独立一级页面优于塞进奇门侧栏或八字结果底部。
- 宽屏使用左侧筛选、中央列表、右侧预览；窄桌面退化为列表 + 详情抽屉。
- 支持键盘聚焦、Enter 打开、上下移动、Esc 关闭菜单；危险操作不能绑定单击快捷键。

### 7.4 一致性

奇门和八字共用按钮尺寸、状态颜色、错误/撤销反馈、记录菜单和 AI 卡片结构；不强求相同的盘面网格、术语、专业信息顺序或主色占比。

### 8.9 跨来源归纳

在本轮样本中反复出现、可以谨慎称为“专业产品常见方式”的设计模式：

1. 历法、地点、真太阳时和时间边界显式输入，并在结果回显采用值。
2. 基础盘、标准盘和专业细盘分层，但来自同一计算结果。
3. 原局、大运、流年、流月是连续的选择链，而不是互不关联的长表。
4. 奇门通过宫位聚焦、八字通过密度或层级切换处理手机信息量。
5. 记录管理是一级能力，常见分类、姓名/日期排序、笔记和备份。
6. 显示偏好与计算规则分开：隐藏神煞、颜色和密度是视图；真太阳时、换日和流派属于计算口径。

值得借鉴的是这些信息组织原则；不适合当前项目的是云账号、会员、社区、课程、古籍库、自动收集案例、任意流派切换、无限显示设置和未经解释的评分图表。

## 9. 从零设计的理想版本

这是一套不受当前页面布局限制的产品蓝图，不是要求重写网站。实施时仍应渐进改造并复用现有正确结果。

### 9.1 设计原则

1. **盘面先于解释。** 先显示采用值和盘面身份，再显示规则推导，最后进入 AI。
2. **一个工作区只围绕一个明确对象。** 奇门是一盘一事件；八字是一命盘、多主题会话。
3. **专业深度按需展开。** 普通与专业差异是显示深度，不是两套算法。
4. **记录是一级页面。** 保存记录不是结果页底部的附属列表。
5. **本地优先但不隐瞒边界。** 明确仅此浏览器，提供可恢复备份。

### 9.2 核心页面决策

| 页面/场景 | 当前问题 | 用户影响 | 理想方案 | 手机端 | 桌面端 | 风险 | 对排盘/AI/历史的影响 | 优先级 |
|---|---|---|---|---|---|---|---|---|
| 首页/工作台 | 奇门、八字和记录入口缺少统一的继续工作视角 | 高频用户每次都像重新开始 | 展示“新建奇门、新建八字、继续当前盘、最近使用、本机数据状态” | 单列卡片，最近 3–5 条 | 双模块快捷入口 + 最近记录 | 低至中 | 排盘不改；AI不改；历史只读 | P2 |
| 起盘/出生输入 | 输入、默认偏好和实际采用值混在一起 | 校盘困难、默认状态可能被误解 | 分成“必要输入、时间/地点校正、可选人物/问题”，提交前显示采用值预览 | 分步但不强制多页；高级项折叠 | 单页分组，右侧实时回执 | 中 | 排盘不改；AI仅人物/问题上下文；历史不改 | P1 |
| 奇门盘面工作区 | 九宫、详情、AI、记录在长页中上下分散 | 点击反馈远、状态易漂移 | 九宫为中心；身份条、选中宫位、AI 和保存状态围绕同一 chartId | “盘面/宫位/AI”可见标签，宫位底部抽屉 | 主盘 + 右侧上下文 | 中 | 排盘不改；AI绑定盘面；历史兼容 | P1 |
| 八字盘面工作区 | 顶部信息重复，主标签过晚，大运表过密 | 查找慢、手机难读 | 紧凑四柱 + “原局/格局喜用/大运流年/AI”；运岁逐层进入 | 吸顶标签、局部滚动、当前链固定 | 概览+依据双栏、完整表 | 中 | 排盘不改；AI可显式选运；历史兼容 | P1 |

### 9.4 奇门与八字切换

- 顶部始终有“奇门 / 八字”模块切换，但切换时不丢当前工作区。
- 若当前盘有未保存修改，先显示“保留草稿并切换 / 取消”，不静默丢弃。
- 记录中心可以同时搜索两类记录，打开后进入各自工作区。
- 不能把奇门宫位和八字运岁抽象成同一交互，也不能把两套 AI 上下文混在一起。

### 9.5 一次完整使用流程

#### 奇门

1. 新建奇门，默认当前时间；明确显示普通时间或真太阳时。
2. 输入原始问事和问事类型，可选姓名/性别；预览最终采用时间、经度和四柱。
3. 起盘后锁定原始问事和盘面身份；修改计算输入即进入“需重新起盘”。
4. 查看九宫，点击宫位即时看到详情。
5. 用户主动开始 AI；回答始终标注当前盘和原始问事。
6. 继续追问只进入本事件会话；新事件明确新起盘。
7. 保存时设置标题、人物、标签和笔记；以后选择更新或另存。
8. 记录重开后恢复盘面、对话、笔记、选中宫位和阅读位置。

#### 八字

1. 输入姓名、性别、出生时间、地点与真太阳时。
2. 提交前回显输入时间、最终采用时间、经度和边界风险。
3. 进入四柱概览，默认先看核心结构，专业依据按需展开。
4. 在原局、格局喜用、大运流年之间切换；当前运→年→月链始终可见。
5. 首次 AI 建立主题会话；事业、感情或某年运势可另建会话。
6. 保存时区分元数据编辑和出生信息变化；后者默认另存副本。
7. 按姓名、标题、日期、四柱、拼音或标签重新打开。

### 9.6 数百条记录后的工作方式

- 默认进入最近使用；姓名排序时才启用 A–Z。
- 记录列表只展示稳定事实，点击进入完整工作区。
- 专业用户可给记录添加不进入 AI 的个人笔记、待复核项和标签。
- 多选首批只做批量标签和导出；删除必须有撤销/备份。
- 奇门与八字可手工关联到同一人物，但关联只用于导航。

### 9.7 AI 首次分析和继续追问

- AI 不是独立于盘面的聊天页，而是当前盘的上下文面板。
- 每个回答头部显示“第几问、完整日期、使用的盘面/运岁快照、状态”。
- 默认展开最新回答，旧回答折叠；提供一键收起、复制、重试和跳到最新。
- 奇门首次问事固定；八字允许主题会话，但新会话默认不带全部旧对话。
- 用户笔记、标签、收藏和关联记录默认不进入 AI。

### 9.8 专业信息分层

## 10. 当前版本与理想版本差异矩阵

“专业网站常见方式”仅指第 8 节样本中可公开确认、且多次出现的方式，不代表全行业统一标准。

| 功能或流程 | 当前项目 | 专业网站常见方式 | 理想方案 | 是否值得改 | 风险 | 优先级 |
|---|---|---|---|---|---|---|
| 奇门九宫主视觉 | 视觉辨识度高，手机 390px 基本可读 | 保持九宫紧凑，点击宫位看详情 | 保留现有盘面视觉，只调整周边层级 | 保留为主，不重做 | 低 | 保留 |

| AI 请求生命周期 | 可能跨盘返回或重复发送 | 公开样本强调锁定盘面，但异常实现未知 | requestId + chartId + recordId，取消/重试 | 必须 | 中高 | P0 |

### 10.1 当前明显领先或应保留

- 奇门九宫盘的视觉辨识度和 390px 下的基本可读性。
- 奇门当前/前后时辰、真太阳时预览、问事类型和按问折叠/删除。
- 八字真太阳时修正说明、格局证据和不确定性表达。
- 两模块都已具备完整可工作的本地 AI 主流程，不需要为了“像专业站”改写命理结果。

### 10.2 明显落后且值得快速补齐

- 盘面脏状态、AI 请求归属和活动记录状态。
- localStorage 损坏/配额/静默截断保护。
- 八字返回当前盘、吸顶标签和手机表格。
- 奇门平板/320px 响应式、宫位详情邻近反馈。
- 独立记录中心的搜索、排序和稳定卡片。

### 10.3 需要以后结构调整

- 八字多主题 AI 会话和上下文裁剪。
- 旁路元数据索引、数百条虚拟列表及可选 IndexedDB 适配层。
- 历史结果版本追溯、关联记录、专业显示密度。

## 11. P0/P1/P2 优先级实施路线

### 11.1 第 0 阶段：先建立验证护栏

| 工作 | 完成标准 | 风险/边界 |
|---|---|---|
| 固定现有奇门和八字代表性基线 | UI 改造前后同一输入的盘面核心字段完全一致 | 不修改规则期望值以迎合 UI |
| 建立历史记录夹具 | 旧奇门自动/正式记录和旧八字记录均可打开 | 不批量重写真实 localStorage |
| 覆盖手机 320/390、平板约 1024、桌面 1440 | 无整页横向溢出，关键动作可达 | 视觉快照不能替代理论基准 |

### 11.2 P0-A：盘面、AI 和记录身份

1. P0-01、P0-09：奇门计算输入、默认偏好、盘面指纹和 dirty 门禁。
2. P0-02、P1-15：性别、地点和实际经度一致恢复。
3. P0-10：原始问事锁定，追问不覆盖记录主题。
4. P0-05：所有 AI 请求绑定盘面/记录/请求 ID。
5. P0-03、P0-04、P0-06：活动记录状态、更新/另存和返回当前盘。

验收重点：不改变任何盘面结果；旧记录照常打开；迟到 AI 回答不能进入另一盘。

### 11.3 P0-B：数据保护

1. 去除八字静默 50 条截断。
2. 区分“无记录、解析损坏、配额不足、存储不可用”。
3. 保存失败不得清空当前盘或覆盖损坏原文。
4. 先提供原始/结构化导出，再增加带预览的恢复。

在这一阶段完成前，不开发批量删除、完整回收站或自动迁移。

### 11.4 P1-A：高频导航与手机可用性

1. 八字“返回当前盘”、吸顶主标签和紧凑概览。
2. 奇门 320px 与 901–1085px 布局修复，宫位详情移近盘面。
3. 44px 触控目标、主次操作和错误/撤销反馈。
4. 八字专业表格局部滚动与当前列定位。
5. 两模块统一 AI 作曲器、草稿和新回答锚点。

### 11.5 P1-B：记录中心第一版

依赖 P0-B 完成后再做：

1. 兼容读取适配层和统一 `uid`；
2. 姓名、标题、日期、四柱、问题和拼音搜索；
3. 最近使用、创建时间、姓名、盘面日期排序；
4. 模块/来源/问事类型/标签筛选；
5. 清晰卡片、空状态、无结果状态和重复保存提醒；
6. 手机全屏记录页、A–Z、返回位置恢复。

### 11.6 P2：专业效率

- 用户笔记/待复核、收藏、批量标签/导出；
- 完整时间戳、显示密度、可访问性和匿名导出；
- 版本来源、关联记录、八字主题会话；
- 单盘稳定和数据容量评估后，再讨论两盘对照。

## 12. 低风险高收益的第一批建议

以下顺序不是直接开工授权，而是用户确认后最适合拆成小提交的第一批：

| 顺序 | 建议 | 收益 | 风险 | 依赖/验证 |
|---|---|---|---|---|
| 1 | 修复奇门真太阳时偏好回显、地点/经度标签，并在输入变化时显示“需重新起盘” | 立即消除最危险的状态误导 | 低至中 | 同一输入前后盘面核心字段不变 |
| 2 | 增加盘面身份条和活动记录状态：未保存、已保存、有修改、保存失败 | 保存与 AI 归属一眼可见 | 中 | 旧记录打开、更新、另存的兼容测试 |
| 3 | AI 请求绑定当前盘并防重复发送 | 消除跨盘串答和重复请求 | 中 | 延迟返回、切盘、取消、重试测试 |
| 4 | 修复八字返回当前盘、吸顶标签和奇门关键响应式断点/触控尺寸 | 手机和桌面高频路径明显变短 | 低至中 | 320/390/1024/1440 回归 |
| 5 | 数据保护底线：取消 50 条截断、解析/配额错误分型、原始备份导出 | 防止长期案例静默丢失 | 中 | 损坏 JSON、空间不足、51+ 条记录测试 |
| 6 | 统一一个 AI 作曲器，长依据默认折叠，新回答自动定位且保留草稿 | 长对话立即更易读 | 中 | 旧问答渲染、折叠、删除、草稿测试 |
| 7 | 记录中心 v1：搜索、最近使用/创建时间/姓名排序和卡片 | 数十条记录即有明显收益 | 中 | 必须在数据保护后；不迁移旧键 |

若只能先选五项，应选表中的 1–5；记录中心虽很重要，但必须建立在不丢数据和稳定记录身份之上。

## 13. 暂时不建议实施的内容

| 内容 | 暂缓原因 |
|---|---|
| 登录、云同步、会员、商业化和多人协作 | 超出自用项目目标，引入隐私、账号和运维复杂度 |
| 一次性重写整个 UI | 难验证状态与历史兼容，且会掩盖算法回归 |
| 直接把三份 localStorage 合并或批量迁移 | 风险高；先做兼容读取层和可恢复备份 |
| 自动按姓名合并人物档案 | 同名、别名和标题混用会误合并 |
| AI 自动改标题、打标签、总结旧记录 | 可能错误归类，也扩大敏感数据处理 |
| 默认搜索完整 AI 正文 | 噪声、隐私和索引体积都高 |
| 完整永久回收站 | 超长 AI 数据会加速容量耗尽；先做短时撤销 |
| 批量删除 | 数据保护和撤销未成熟前风险过高 |
| 多盘并排/任意多盘对照 | 单盘身份、手机布局和记录可靠性尚未稳定 |
| 盘面画笔、复杂标注和教学演示模式 | 触控冲突与实现成本高，优先级低于搜索和备份 |
| 新增奇门流派、八字规则口径或大量算法设置 | 直接扩大排盘逻辑范围并增加验证成本 |
| 根据真人经历反推时辰或反向制定规则 | 不符合规则约束，容易制造过拟合和伪精确 |
| 未有稳定指标的人生 K 线、综合分或游戏化图表 | 视觉权威感高于实际可解释性 |
| 古籍库、课程、社区、名人库和每日组件 | 范围扩张、版权和维护成本，与当前核心目标无关 |

## 14. 风险、数据兼容性和排盘逻辑保护说明

### 14.1 三条硬边界

1. UX 改造只消费现有排盘结果，不在展示层重算四柱、局式、格局、喜用或运岁。
2. 任何新增标签、收藏、笔记、关联、排序和最近使用信息默认不进入 AI。
3. 旧记录可缺少新字段，但必须继续打开；禁止为“整洁”而在启动时批量覆盖。

### 14.2 变更类型与风险

| 类型 | 示例 | 排盘逻辑 | AI 上下文 | 历史兼容 |
|---|---|---|---|---|
| 纯展示 | 折叠、吸顶标签、字号、触控尺寸 | 不影响 | 不影响 | 不影响 |
| 状态保护 | dirty、盘面指纹、请求绑定 | 不影响计算结果 | 防止错盘；请求身份会变化 | 可只存在内存或增量字段 |
| AI 上下文接线 | 当前运岁、原始问事锁定、主题会话 | 不影响 | 明确改变输入组织，必须单独评审 | 旧回答不重写 |
| 旁路元数据 | 标签、收藏、最近使用、拼音覆盖 | 不影响 | 默认禁止进入 | 原键不动，可重建 |
| 数据保护 | 取消截断、解析/配额处理、备份 | 不影响 | 不影响推理 | 必须以非破坏读取为前提 |
| 结构迁移 | IndexedDB、多会话、版本快照 | 不影响目标，但实现风险高 | 可能改变上下文范围 | 必须后做、可回滚 |

### 14.3 localStorage 兼容策略

- `qimen_auto_history`、`qimen_history`、`bazi_records` 继续作为原始事实来源。
- 新字段可选、缺失可回退；数字/字符串原始 ID 原样保留。
- 旁路索引可随时重建，不能成为打开盘面的唯一来源。
- 导入前预览数量、版本和冲突；默认合并或另存，不静默覆盖。
- API Key 永不进入记录备份、匿名报告或搜索索引。
- 若以后容量确实超过 localStorage，应先做双读兼容和完整备份，再讨论 IndexedDB，不能一次切换。

### 14.4 AI 上下文保护

- 每次请求保存或内存绑定盘面指纹、记录 ID、会话 ID 和运岁选择快照。
- UI 显示的当前盘必须与请求绑定值一致；不一致就拒绝发送。
- 奇门原始问事固定；八字新主题默认不带全部旧对话。
- 笔记、生平事实、标签、收藏和关联记录只有用户明确选择才可加入一次请求。
- AI 失败不得覆盖已成功回答；重试要保留原问题和请求来源。

### 14.5 每批实施的建议检查

- 奇门、八字模块规定的现有语法、页面和理论基准检查全部通过。
- 代表性输入的盘面核心字段改造前后逐项一致。
- 旧记录、缺字段记录、损坏 JSON、51+ 八字记录、空间不足分别覆盖。
- AI 延迟返回、连续点击、切盘、删除当前记录和刷新恢复分别覆盖。
- 320、390、768/1024、1440px 无整页横向滚动；关键按钮可触达。
- 键盘标签页、宫位焦点、屏幕阅读器名称和高对比状态做基础检查。

## 15. 待用户确认的问题

以下问题会影响后续实施取舍，本报告不替用户做超范围决定：

1. **第一批是否按第 12 节 1–5 顺序实施？** 推荐先状态正确性和数据保护，再做记录中心。
2. **正式记录修改排盘输入时，默认行为是否统一为“另存副本”？** 推荐是；标题/标签等元数据仍可更新原记录。
3. **记录中心是否先做统一界面、底层继续保持三份存储？** 推荐是，兼容风险最低。
4. **八字是否需要“一个命盘多个 AI 主题会话”？** 推荐列为 P2，先测量现有长会话的实际数量和上下文问题。
5. **个人笔记和待复核项是否进入首个记录中心版本？** 推荐先做基础笔记，但默认永不进入 AI。
6. **姓名 A–Z 是否只在姓名排序时显示？** 推荐是，避免与“最近使用”排序冲突。
7. **删除保护采用短时撤销还是完整回收站？** 推荐先做 8–10 秒撤销，等容量方案稳定后再评估回收站。
8. **简洁/标准/专业显示密度是否需要首批？** 推荐暂放 P2，先把默认标准模式层级做好。
9. **是否需要手工关联同一人物的奇门和八字记录？** 推荐 P2，只做导航关系，不自动合并、不进入 AI。
10. **手机静态页更新提示是否值得做？** 若经常跨设备验证 GitHub Pages，推荐在状态/草稿保护之后加入轻量版本提示。
