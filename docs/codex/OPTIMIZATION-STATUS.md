## 状态口径

| 状态 | 判定标准 |
| --- | --- |
| 已完成 | 已合入 `main`，并有代码、测试或可复算报告作为证据 |
| 部分完成 | 已实现部分入口或护栏，但原设计仍有明确缺口 |
| 未开始 | 当前 `main` 未找到可靠实现证据 |
| 进行中 | 已有首期实现，规则库或覆盖范围仍持续扩展 |
| 暂缓 | 已明确后做，不能顺手扩大范围 |
| 不采用 | 已确认当前阶段不实施，并保留原因 |

状态冲突时按以下顺序裁决：`main` 代码与自动测试 > `PROJECT-STATUS.md` > 本总表 > 各历史设计文档。历史文档不删除旧结论，只补充当前状态和版本。

## 维护规则

1. 每次开始优化前，先确认对应 `P*-*` 或 `PAT-*` 编号。
2. 代码完成后必须先通过相关测试，再把状态改为“已完成”。
3. 只写了方案或只在独立工作树实现，不得标为共享主线已完成。
4. 同一项只完成一部分时，必须写清“已完成内容”和“剩余缺口”。
5. 规则、评分模型或理论基准变化时，同时更新设计文档、基准版本和本总表。
6. 整体 UI/产品状态更新到审查索引；格局喜用细项更新到 `PAT-*` 清单；本总表只保留跨文档摘要。
